# Feishu Custom Refactor Design

## Goal

Refactor `feishu-custom` from an all-in-one script into a layered system:

1. **Official Feishu tools / CLI / API** for auth, token lifecycle, transport, standard doc create/fetch/update
2. **Deterministic preprocess layer** for formatting contract generation
3. **Optional LLM semantic assist layer** for content classification / enhancement
4. **Regression + verification layer** for output fidelity

The core idea is:

- move boring infrastructure to official tooling
- keep contract-critical rendering logic local and deterministic
- prefer create-time rich text expression over post-patch when possible

---

## High-Level Conclusion

### What can move to official tools

- auth / OAuth / token lifecycle
- API transport
- document create / fetch / update
- wiki / drive standard operations
- markdown features already supported by Lark-flavored markdown

### What should stay local

- heading numbering generation
- title vs body-H1 contract
- numbering color contract
- deterministic preprocess rules
- regression fixtures / normalization / verification
- rare fallback patch logic

### Most important finding from experiments

**Prefix-only blue numbering can be expressed at create time without post-patch**, using Lark rich text syntax:

```md
# <text color="blue">1. </text>Comprehensive Feishu Hybrid Regression
```

This worked.

The following did **not** work reliably:

```md
# <span style="color: blue">1. </span>Comprehensive Feishu Hybrid Regression
```

Therefore:

- use Feishu-native rich text tags
- do not rely on raw HTML span styling in headings

## Verified Findings

### 1. Native heading auto-numbering is not available through normal markdown import

Observed behavior:

```md
# Comprehensive Feishu Hybrid Regression
## Background
## Scope
```

Result:
- headings stay plain headings
- no automatic numbering appeared

Implication:
- numbering must still be generated by preprocess logic

### 2. Manual numbering is treated as heading text

Observed behavior:

```md
# 1. Comprehensive Feishu Hybrid Regression
## 1.1 Background
```

Result:
- numbering persisted as plain heading text
- no evidence of native numbering semantics

Implication:
- use deterministic numbering generation in preprocess
- do not depend on native numbering state

### 3. Whole heading blue works at create time

Working form:

```md
# <text color="blue">1. Comprehensive Feishu Hybrid Regression</text>
```

### 4. Prefix-only blue works at create time

Working form:

```md
# <text color="blue">1. </text>Comprehensive Feishu Hybrid Regression
```

### 5. HTML span-based inline styling in heading did not hold

Rejected form:

```md
# <span style="color: blue">1. </span>Comprehensive Feishu Hybrid Regression
```

### 6. Heading text-level background color works

Working form:

```md
## <text bgcolor="yellow">Heading yellow bg</text>
## <text bgcolor="green">Heading green bg</text>
```

Verified by fetch-back: `bgcolor` attribute is preserved in roundtrip.

### 7. Blue numbering + colored body text works

Working form:

```md
## <text color="blue">1.1 </text><text color="red">Red Title</text>
## <text color="blue">1.1 </text><text color="purple">Purple Title</text>
```

### 8. Blue numbering + heading background works

Working form:

```md
## <text color="blue">1.1 </text><text bgcolor="yellow">Yellow bg Title</text>
```

### 9. Callout light backgrounds work

Working form:

```md
<callout emoji="💡" background-color="light-blue" border-color="blue">content</callout>
<callout emoji="⚠️" background-color="light-yellow" border-color="yellow">content</callout>
<callout emoji="✅" background-color="light-green" border-color="green">content</callout>
```

Supported light colors: light-red/red, light-blue/blue, light-green/green, light-yellow/yellow, light-orange/orange, light-purple/purple, pale-gray/light-gray/dark-gray.

### 10. Inline text colors and backgrounds work

```md
<text color="red">red</text> <text color="blue">blue</text> <text color="green">green</text>
<text bgcolor="yellow">yellow bg</text> <text bgcolor="green">green bg</text> <text bgcolor="orange">orange bg</text>
```

### 11. CLI can call block batch_update API

Source code confirmed: `larksuite/cli` uses `/open-apis/docx/v1/documents/:id/blocks/batch_update` for media operations.
This proves the CLI has block-level API access, but no ready-made shortcut exists for heading background color.

### 12. Official CLI is useful for auth + standard flows, but heading background color has no ready-made create-time syntax

We successfully validated:
- CLI install
- app init
- user login
- doc fetch
- block listing
- text-level background in headings (`<text bgcolor="...">`)
- callout light backgrounds
- rainbow text colors

Not found in official reference:
- block-level heading background color
- `{background-color="..."}` on headings

## Architecture

## Layer 1 — Official Tooling Layer

Responsible for:
- auth
- token lifecycle
- doc create / fetch / update
- wiki / drive standard access
- standard media operations

This layer should contain as little business logic as possible.

### Inputs
- already-preprocessed Lark-flavored markdown
- target metadata (title, folder/wiki target)

### Outputs
- document IDs / URLs
- fetched markdown for verification
- API errors / transport logs

---

## Layer 2 — Deterministic Preprocess Layer

This is the core of the system.

Responsible for:
- heading numbering generation
- title vs first H1 decision
- heading color insertion
- prefix-only blue numbering generation
- markdown normalization
- table / structural normalization where needed
- rule-based mapping to Lark markdown constructs

### Why deterministic

Anything that affects regression stability should not be left to probabilistic generation.

This includes:
- numbering
- exact heading text decoration
- exact tag wrapping
- contract-critical syntax transforms

### Core output style examples

#### Whole heading blue

```md
# <text color="blue">1. Comprehensive Feishu Hybrid Regression</text>
```

#### Prefix-only blue

```md
# <text color="blue">1. </text>Comprehensive Feishu Hybrid Regression
## <text color="blue">1.1 </text>Background
### <text color="blue">1.1.1 </text>Problem Statement
```

### Recommended preprocess responsibilities

1. Detect whether document title duplicates first H1
2. Decide whether first heading should be promoted to document title
3. Generate numbering sequence
4. Wrap numbering prefix in `<text color="blue">...</text>` when contract requires blue numbering
5. Leave heading body unwrapped unless whole-heading color is intended
6. Emit valid Lark-flavored markdown only

---

## Layer 3 — LLM Assist Layer

Use LLM only for semantic judgment, not exact formatting.

### Good uses
- decide whether a paragraph should become a heading
- decide whether content should become callout / note / quote
- improve phrasing of headings
- infer section hierarchy from raw notes

### Bad uses
- exact numbering
- exact color wrapping
- exact title/H1 contract formatting
- anything that must be stable across runs

### Rule

**LLM proposes, script enforces.**

Suggested workflow:

1. LLM outputs semantic structure plan
2. deterministic preprocess converts plan into exact Lark markdown

---

## Layer 4 — Regression and Verification Layer

Responsible for:
- fetch-back verification
- normalization
- fixture comparison
- render-contract checks
- large-table sanity checks

### Minimum verification checklist

1. count-sensitive checks when needed
2. duplicate-sensitive checks when needed
3. required-field completeness
4. heading structure validation
5. title/H1 contract validation
6. color-tag validation for numbered headings
7. fetch-back roundtrip checks on representative cases

### Roundtrip expectation

For prefix-only blue numbering, fetch-back should preserve forms like:

```md
# <text color="blue">1. </text>Comprehensive Feishu Hybrid Regression
```

If fetch-back loses these wrappers, the transform is not reliable enough.

## Reference Syntax Guide

## Headings

### Plain heading

```md
# Title
## Section
```

### Whole heading blue

```md
# <text color="blue">1. Title</text>
```

### Prefix-only blue

```md
# <text color="blue">1. </text>Title
## <text color="blue">1.1 </text>Section
### <text color="blue">1.1.1 </text>Subsection
```

### Avoid

```md
# <span style="color: blue">1. </span>Title
```

because HTML span styling did not survive reliably.

## Title vs H1 contract

### Recommended rule

If the first H1 is semantically the document title, move it to document title and remove duplicate body H1.

If the first H1 is a real section heading, keep it as H1 and number it.

### Preferred default

- document title holds the document title
- body starts from first true section heading

## Background color / callout / block styling

Use Lark-native markdown constructs whenever possible.

Examples to prefer:
- `<text color="blue">...</text>` for inline text color
- callout syntax / Lark-supported structured tags for block emphasis

Avoid depending on raw HTML/CSS styling if a Lark-native form exists.

Because patch-level proof is still weak, treat **create-time native markup** as primary and **block patch** as fallback.

---

## Recommended Decision Matrix

### Use preprocess script when
- formatting must be exact
- output must be regression-stable
- the rule is deterministic
- numbering is involved
- color wrapping is contract-critical

### Use LLM when
- semantic classification is needed
- sectioning is ambiguous
- tone/phrasing improvement helps
- content structure needs interpretation

### Use official tooling directly when
- auth / transport only
- creating/updating docs from already-correct markdown
- fetching docs for verification
- standard drive/wiki operations

### Use patch fallback only when
- create-time expression cannot represent the requirement
- the effect is impossible or too verbose in markdown
- the target area is small and test-covered

## Migration Plan

## Phase 1 — Infrastructure swap

Replace local auth/transport with official tooling.

Deliverables:
- official auth flow
- standard create/fetch/update wrapper
- stable environment config

## Phase 2 — Preprocess extraction

Extract deterministic preprocess into explicit module(s):
- title/H1 resolution
- numbering generation
- prefix-only blue wrapping
- structure normalization

## Phase 3 — LLM contract boundary

Define exactly what the LLM may output:
- semantic structure hints only
- no direct final formatting unless post-normalized

## Phase 4 — Regression harness

Create fixtures for:
- plain headings
- whole-blue heading
- prefix-only blue heading
- title-only vs H1 cases
- large document representative cases

## Phase 5 — Minimize patch fallback

Only keep patch logic for effects that cannot be represented in create-time markdown.

## Implementation Guidance

## Suggested module split

- `preprocess/structure.py`
  - title/H1 resolution
  - heading tree normalization

- `preprocess/numbering.py`
  - numbering generation
  - prefix extraction

- `preprocess/lark_format.py`
  - wrap prefix with `<text color="blue">...</text>`
  - Lark-specific syntax helpers

- `pipeline/render.py`
  - orchestrate preprocess → official doc create/update

- `verify/fetchback.py`
  - fetch doc markdown back

- `verify/contracts.py`
  - validate title/H1 / numbering / color-tag contract

## Pseudocode for prefix-only blue heading

```python
def render_numbered_heading(level: int, number: str, title: str) -> str:
    prefix = f'<text color="blue">{number} </text>'
    hashes = '#' * level
    return f'{hashes} {prefix}{title}'
```

Where `number` already includes the trailing dot when appropriate, e.g.:
- `1.`
- `1.1`
- `1.1.1`

You may also choose to include the following-space inside the blue tag for consistency:

```python
prefix = f'<text color="blue">{number} </text>'
```

That is the experimentally validated style.

---

## Final Recommendation

Do **not** rewrite `feishu-custom` as a thin wrapper around official CLI only.

Instead:

- use official tooling as the infrastructure substrate
- move contract-critical rendering into deterministic preprocess
- keep LLM only for semantic planning
- use verification aggressively
- treat patch as rare fallback, not default architecture

This gives the best mix of:
- lower maintenance burden
- fewer auth / token headaches
- stable rendering contract
- easier regression control
